**Виконав:** Михальчук Андрій

**Група:** КН-21

**Заклад:** Звягельський політехнічний фаховий коледж (ЗПФК)

## 1. Що таке наслідування? Переваги та недоліки

Наслідування — це механізм ООП, коли один клас (похідний) отримує поля та методи іншого класу (базового) і може їх розширювати або змінювати. У нашому проєкті `Student` успадковує `UniversityMember`, який у свою чергу успадковує `Person`.

Переваги: повторне використання коду (ПІБ, email, вік винесені в `Person`), логічна ієрархія (від особи до студента), зручне розширення новими типами (`ForeignStudent`, `WorkingStudent`). Недоліки: жорсткий зв'язок між класами, складність при глибоких ієрархіях, ризик порушення принципу підстановки Лісков, якщо похідний клас змінює поведінку непередбачувано.

## 2. Приклади використання `base` у коді

У конструкторі `Student` викликається `base(fullName, dateOfBirth, personalEmail, notes)` — ініціалізація полів базового класу `Person`. У `CalculateScholarship()` класу `ExcellentStudent` використовується `base.CalculateScholarship()` для отримання базової стипендії, якщо бал нижче 90. У `GetInfo()` метод `Student` викликає `base.GetInfo()` з `UniversityMember`, який у свою чергу викликає `base.GetInfo()` з `Person` — ланцюжок доповнення інформації.

## 3. Різниця між `virtual` / `override` та `new`

`virtual` у базовому класі дозволяє похідному класу перевизначити метод через `override`. При виклику через змінну базового типу виконується версія реального об'єкта (поліморфізм). Наприклад, `virtual string GetInfo()` у `Person` перевизначено в `Student` через `override`.

Ключове слово `new` приховує метод базового класу, але не забезпечує поліморфізм — при виклику через змінну базового типу виконається метод базового класу, а не похідного. Тому для ієрархії студентів ми використовуємо `override`, а не `new`.

## 4. Для чого потрібні абстрактні класи?

Абстрактний клас задає спільний каркас для групи пов'язаних типів, але не може бути створений напряму. `Person` і `UniversityMember` визначають спільні властивості та методи. `UniversityMember` містить абстрактний `CalculateScholarship()` — кожен тип студента зобов'язаний реалізувати власну логіку. `UniversityEvent` з абстрактним `ConductEvent()` змушує кожен захід мати унікальну поведінку проведення.

## 5. Навіщо використовують `sealed` класи?

`sealed` забороняє подальше наслідування класу. `GraduateStudent` позначено як `sealed`, бо він є кінцевим рівнем ієрархії студентів — випускник не повинен мати додаткових похідних типів. Це захищає логіку від небажаних розширень і допомагає компілятору оптимізувати виклики методів.

## 6. Як реалізовано поліморфізм у системі?

Колекція `List<UniversityMember> _members` зберігає різні типи студентів. При виклику `member.GetInfo()` або `member.CalculateScholarship()` виконується реалізація конкретного класу (`ExcellentStudent`, `ForeignStudent` тощо). Метод `GetMembersByType<T>()` використовує `OfType<T>()` для фільтрації. Заходи `UniversityEvent` демонструють поліморфізм через `ConductEvent()` — один список подій, різна поведінка для хакатону, кібертурніру та конференції.

## 7. Труднощі при проектуванні ієрархії

Основна труднощ — визначити, які поля залишити в базовому класі, а які — у `Student`. Наприклад, `RecordBookNumber` стосується лише студентів, тому він у `Student`, а не в `UniversityMember`. Також потрібно було узгодити конструктори: параметризований конструктор `Student` викликає `base(...)`, а безпараметровий залишено для сумісності. При фільтрації `GetMembersByType<Student>()` повертаються всі студенти, включно з похідними типами, оскільки `OfType<Student>()` включає нащадків.

## 8. Що нового дізналися

Дізналися, як будувати багаторівневу ієрархію класів з абстрактними методами, як правильно використовувати `base` для делегування базовій реалізації, і як поліморфна колекція спрощує роботу з різними типами студентів. Також закріпили практику `sealed` для кінцевих класів і застосували поліморфізм у власному варіанті — системі студентських заходів `UniversityEvent` з хакатоном, кібертурніром CS2 та науковою конференцією.
