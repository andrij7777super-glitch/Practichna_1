## 1. Що таке інкапсуляція?

**Інкапсуляція** — це принцип ООП, за яким внутрішній стан об'єкта приховується від зовнішнього світу, а доступ до нього здійснюється лише через контрольований публічний інтерфейс (властивості та методи). Це захищає дані від некоректних змін і дозволяє змінювати внутрішню реалізацію без впливу на решту системи.

### П'ять прикладів інкапсуляції з коду

| № | Приклад | Файл | Пояснення |
|---|---------|------|-----------|
| 1 | Приватне поле `_fullName` із публічною властивістю `FullName` | `Student.cs` | Прямий доступ до імені заборонений; зміна можлива лише через setter із валідацією (мінімум 3 слова, 5 символів). |
| 2 | Приватний setter властивості `AverageGrade` | `Student.cs` | Зовнішній код не може довільно змінити середній бал — лише через `UpdateAverageGrade()` або журнал оцінок. |
| 3 | Приватне поле `_subjectGrades` у `GradeJournal` | `GradeJournal.cs` | Словник оцінок недоступний для прямої модифікації; зміни відбуваються через `AddOrUpdateGrade()`. |
| 4 | Приватне поле `_students` у `StudentGroup` | `StudentGroup.cs` | Список студентів керується лише методами `AddStudent()` та `RemoveStudent()`, що запобігає дублюванню записів. |
| 5 | Приватний метод `SetAverageGradeInternal()` | `Student.cs` | Централізована логіка валідації та округлення середнього балу прихована від зовнішніх класів. |

## 2. Порівняння авто-властивостей та властивостей з повною реалізацією

| Критерій | Авто-властивість `{ get; set; }` | Властивість з повною реалізацією |
|----------|----------------------------------|----------------------------------|
| Код | Компактний, мінімальний | Розширений, з логікою в get/set |
| Валідація | Відсутня за замовчуванням | Реалізується в setter/getter |
| Коли використовувати | Прості дані без обмежень | Дані, що потребують перевірки або обчислення |

### Коли обирали кожен варіант у проєкті

**Авто-властивості** застосовано для простих полів без спеціальної логики:

- `Status` — enum із фіксованим набором значень;
- `EnrollmentDate` — дата без додаткової валідації;
- `Notes` — довільний текстовий коментар;
- `GroupName`, `Specialty`, `Course` у класі `StudentGroup`.

**Властивості з повною реалізацією** — там, де потрібен контроль:

- `FullName` — перевірка кількості слів і довжини;
- `PersonalEmail` — regex-валідація формату email;
- `RecordBookNumber` — перевірка на рівно 8 цифр;
- `AverageGrade` — обмеження діапазону 0–100 та округлення до 2 знаків.

## 3. Валідація даних

### Для яких даних застосовано валідацію

| Дані | Де реалізовано | Правило |
|------|----------------|---------|
| ПІБ (`FullName`) | `Student.cs` | Не порожнє, ≥ 5 символів, ≥ 3 слова |
| Номер залікової (`RecordBookNumber`) | `Student.cs` | Рівно 8 цифр (regex `^\d{8}$`) |
| Email (`PersonalEmail`) | `Student.cs` | Формат email (regex) |
| Середній бал (`AverageGrade`) | `Student.cs` | Діапазон 0–100, округлення до 2 знаків |
| Оцінка за предмет | `GradeJournal.cs` | Діапазон 0–100 |
| Назва предмету | `GradeJournal.cs` | Не порожній рядок |
| Унікальність студента | `StudentGroup.cs` | Заборона дублювання `RecordBookNumber` |

### Види валідації

1. **Перевірка на порожність** — `string.IsNullOrWhiteSpace()` для текстових полів.
2. **Regex-валідація** — формат email та номера залікової книжки.
3. **Діапазон значень** — оцінки та середній бал (0–100).
4. **Структурна валідація** — кількість слів у ПІБ.
5. **Бізнес-правила** — унікальність студента в групі, перевірка дати в консольному меню.

При порушенні правил кидаються винятки `ArgumentException`, `ArgumentOutOfRangeException` або `InvalidOperationException`.

## 4. Read-only та init-only властивості

### Навіщо вони потрібні?

- **Read-only (`get` only)** — значення обчислюється динамічно і не повинно зберігатися окремо (уникнення розбіжностей у даних).
- **Init-only (`init`)** — значення встановлюється один раз під час створення об'єкта і не змінюється протягом життєвого циклу (наприклад, дата народження).

### Приклади з коду

```csharp
// Init-only: дата народження не змінюється після ініціалізації
public DateTime DateOfBirth { get; init; }

// Read-only обчислювана: вік завжди актуальний
public int Age => CalculateAge();

// Read-only обчислювана в StudentGroup
public int GroupSize => _students.Count;
public double AverageGroupGrade => ...;
```

`DateOfBirth` обрано як `init`, оскільки дата народження — незмінна характеристика людини. `Age` — read-only, бо вік залежить від поточної дати і не має сенсу зберігати його в полі.

## 5. Властивість Age

Властивість `Age` реалізована як **обчислювана read-only властивість**:

```csharp
public int Age => CalculateAge();

public int CalculateAge()
{
    var today = DateTime.Today;
    var age = today.Year - DateOfBirth.Year;
    if (DateOfBirth.Date > today.AddYears(-age))
        age--;
    return age;
}
```

### Чому вона обчислювана?

1. **Актуальність** — вік змінюється щороку; зберігання в полі призвело б до застарілих даних.
2. **Єдине джерело правди** — вік походить лише з `DateOfBirth`, немає ризику розбіжності.
3. **Принцип DRY** — логіка обчислення зосереджена в одному методі `CalculateAge()`, який використовується і властивістю `Age`, і методом `ShowDetailedInfo()`.

Алгоритм враховує, чи вже відбувся день народження в поточному році (корекція на 1 рік).

## 6. Принцип єдиної відповідальності

Кожен клас відповідає за **одну чітку область** системи:

| Клас | Відповідальність |
|------|------------------|
| `Student` | Представлення окремого студента: особисті дані, статус, вік, академічні показники |
| `GradeJournal` | Зберігання оцінок за предметами та перерахунок середнього балу |
| `StudentGroup` | Управління колекцією студентів: додавання, пошук, статистика, збереження |
| `StudentStatus` | Перелік можливих статусів студента |
| `Program` | Консольний інтерфейс користувача та координація дій |

**Приклад розділення:** клас `Student` не зберігає оцінки напряму — це робить `GradeJournal`. Клас `StudentGroup` не виводить меню — це робить `Program.cs`. Такий поділ спрощує тестування, супровід і подальше розширення системи (наприклад, додавання GUI або Web API).

## 7. Труднощі під час проєктування

| Труднощі | Рішення |
|----------|---------|
| **Зв'язок `AverageGrade` із `GradeJournal`** | Реалізовано патерн зворотного виклику (`Action<double>`): при зміні оцінки журнал автоматично оновлює середній бал студента. |
| **JSON-серіалізація приватних полів** | Використано атрибут `[JsonInclude]` для `_gradeJournal` та `SubjectGrades`; після завантаження викликається `RestoreJournalBinding()` для відновлення зв'язку. |
| **Init-only `DateOfBirth` при десеріалізації** | `System.Text.Json` у .NET 8 підтримує `init`-властивості; об'єкти створюються через `[JsonConstructor]`. |
| **Пагінація в консолі** | Реалізовано навігацію сторінками по 10 записів із командами N/P/D/Q без залежності від сторонніх бібліотек. |
| **Валідація ПІБ** | Узгоджено вимогу «3 слова» з українським форматом «Прізвище Ім'я По батькові» через `Split` із `RemoveEmptyEntries`. |

## 8. Що нового дізналися

1. **Різні типи властивостей C# 11+** — `required`, `init`, read-only обчислювані, властивості з приватним setter у поєднанні в одному класі.
2. **Інтеграція доменних класів через callback** — зв'язок `GradeJournal` → `Student.AverageGrade` без тісного зчеплення.
3. **Серіалізація складних об'єктів** — робота з `[JsonInclude]`, `[JsonIgnore]`, `[JsonConstructor]` для збереження інкапсуляції під час JSON-збереження.
4. **Практичне застосування SRP** — розділення бізнес-логіки (`StudentGroup`) та UI (`Program.cs`).
5. **Консольна пагінація** — реалізація зручної навігації в текстовому інтерфейсі без GUI-фреймворків.
